# Техническое задание
## Приватный чат для общения с друзьями

**Этап 1:** веб-версия / PWA  
**Этап 2:** мобильное приложение

### Назначение документа
Определение функционального объёма MVP, архитектурной схемы и требований к инфраструктуре / железу.

### Целевая аудитория
Заказчик, разработчик, DevOps / системный администратор, тестировщик.

### Формат запуска
Сначала веб-чат через браузер и PWA, затем мобильное приложение на том же backend/API.

### Рекомендуемый базовый стек
- **Frontend:** Next.js
- **Backend/API + realtime:** Node.js/NestJS + Socket.IO либо Supabase
- **БД:** PostgreSQL
- **Хранение файлов:** S3-совместимое хранилище

**Версия:** 1.0  
**Дата:** 10.04.2026

---

## 1. Цель проекта
Создать приватный чат для общения с друзьями с поэтапным запуском: сначала веб-версия, доступная из браузера и устанавливаемая как PWA, затем мобильное приложение на Android / iOS с использованием того же backend, базы данных и модели авторизации.

Проект ориентирован на небольшой закрытый круг пользователей, поэтому основной приоритет — приватность, стабильность, простота использования, умеренная стоимость эксплуатации и возможность быстро запустить MVP без перегруженной архитектуры.

## 2. Объём проекта и этапность

| Этап | Состав работ | Результат |
|---|---|---|
| Этап 1 | Веб-клиент, PWA, backend/API, realtime, БД, хранение файлов, базовая админка, деплой | Работающий чат через браузер и на телефоне как установленная PWA |
| Этап 2 | Нативный/кроссплатформенный мобильный клиент, push-уведомления, публикация сборок, доработка UX | Мобильное приложение на том же backend |

## 3. Основные бизнес-требования
- Сервис должен использоваться только приглашёнными пользователями или пользователями из заранее определённого круга.
- Пользователь должен иметь возможность общаться в личных и групповых чатах.
- История сообщений должна сохраняться и быть доступной после повторного входа.
- Интерфейс должен корректно работать на desktop и mobile-экранах.
- Система должна поддерживать дальнейший перенос фронтенда в мобильное приложение без полной переработки backend-части.
- Эксплуатация должна быть возможна как в облаке, так и на собственном VPS / выделенном сервере.

## 4. Функциональные требования (MVP)

### 4.1 Пользователи и доступ
- Регистрация по email, magic link, одноразовому коду или приглашению.
- Вход и повторная авторизация с безопасным хранением сессии.
- Профиль пользователя: имя, nickname, аватар, краткий статус.
- Возможность заблокировать пользователя или удалить его из приватного пространства администраторами.

### 4.2 Чаты
- Личные чаты 1:1.
- Групповые чаты.
- Создание чата, изменение названия, аватара группы и состава участников.
- Список диалогов с последним сообщением, временем и индикатором непрочитанного.

### 4.3 Сообщения
- Отправка и получение текстовых сообщений в реальном времени.
- Поддержка вложений: изображения, документы в рамках разрешённого формата и размера.
- Статусы доставки и прочтения.
- Редактирование и удаление собственных сообщений в пределах заданного окна или по правилам проекта.
- Подгрузка истории сообщений при прокрутке.

### 4.4 Уведомления и активность
- Индикатор online/offline.
- Индикатор «печатает».
- Web push / browser notifications на веб-этапе.
- Push-уведомления на мобильном этапе.
- Гибкая настройка mute/notification per chat.

### 4.5 Администрирование
- Минимальная админ-панель: список пользователей, список чатов, блокировка пользователя, удаление нежелательного контента.
- Журнал критичных действий администратора.
- Просмотр системного состояния: число активных пользователей, ошибок, объём файлов.

## 5. Нефункциональные требования

| Категория | Требование |
|---|---|
| Производительность | Время открытия списка чатов и диалога — комфортное для пользователя; доставка сообщений в realtime без заметной задержки в типовом режиме |
| Масштабируемость | Архитектура должна позволять вынести realtime/API в отдельный контейнер или сервер без переписывания клиента |
| Безопасность | Передача данных только по HTTPS/WSS; пароли не хранить в открытом виде; обязательная серверная проверка прав доступа к чатам и файлам |
| Резервирование | Регулярный backup базы данных и метаданных; возможность восстановления после сбоя |
| Логирование | Приложение должно писать структурированные логи и метрики по backend, БД и файловому хранилищу |
| Эксплуатация | Развёртывание через Docker Compose на старте; конфигурация через `.env`; единый способ запуска dev/test/prod |

## 6. Рекомендуемая архитектура
Для первого запуска рекомендуется использовать модульную схему, которая не перегружена микросервисами, но уже допускает рост:

- Frontend web/PWA: Next.js
- Backend/API: Node.js / NestJS
- Realtime: Socket.IO или эквивалентный websocket-слой
- Database: PostgreSQL
- Кэш / pub-sub при росте нагрузки: Redis
- Хранение файлов: S3-совместимое object storage (MinIO / cloud object storage)
- Reverse proxy: Nginx / Traefik
- Observability: Grafana + Prometheus + Loki либо внешний managed monitoring

На этапе MVP допускается упрощённый вариант с BaaS/managed-компонентами, однако логическая модель должна сохраняться: frontend ↔ API/realtime ↔ БД/файлы.

## 7. Логическая модель данных

| Сущность | Назначение | Ключевые поля |
|---|---|---|
| `users` | Пользователи системы | `id`, `email/login`, `display_name`, `avatar_url`, `status`, `created_at` |
| `friendships / invites` | Инвайты и связи между пользователями | `id`, `from_user`, `to_user`, `status`, `created_at` |
| `chats` | Карточка чата | `id`, `type(1:1/group)`, `title`, `avatar_url`, `created_by` |
| `chat_members` | Состав участников чата | `chat_id`, `user_id`, `role`, `joined_at` |
| `messages` | Сообщения | `id`, `chat_id`, `sender_id`, `body`, `message_type`, `created_at`, `edited_at` |
| `attachments` | Вложения | `id`, `message_id`, `file_url`, `mime_type`, `size` |
| `message_reads` | Факты прочтения | `message_id`, `user_id`, `read_at` |
| `audit_log` | Системный и админский аудит | `id`, `actor`, `action`, `target`, `created_at` |

## 8. API / backend-требования
- REST API для большинства CRUD-операций и авторизации.
- WebSocket-канал для realtime-сообщений, статуса пользователя, typing-событий и обновления счётчиков непрочитанного.
- Валидация всех входящих данных на сервере.
- Пагинация списков и истории сообщений.
- Ограничение размера файла и белый список допустимых MIME-типов.
- Защита от спама / flood control / rate limiting на публичных точках входа.

## 9. Требования к безопасности
- HTTPS/WSS обязателен для всего внешнего трафика.
- Секреты, ключи и строки подключения должны храниться вне исходного кода.
- Доступ к сообщениям и файлам разрешён только участникам соответствующего чата.
- Должны вестись логи входа, попыток ошибки авторизации и критичных административных действий.
- Необходимо предусмотреть механизм принудительного logout и блокировки пользователя.
- Бэкапы должны храниться отдельно от основного runtime-сервера.

## 10. Требования к инфраструктуре и железу
Ниже приведён практичный sizing для self-hosted-развёртывания. Если используется managed cloud (например, managed PostgreSQL и object storage), требования к собственному серверу можно уменьшить.

### 10.1 Минимальный контур для разработки / пилота

| Контур | CPU | RAM | Disk | Комментарий |
|---|---:|---:|---:|---|
| Dev local / тест | 4 vCPU | 8 GB | 80–120 GB SSD | Достаточно для разработчика, одного backend, Postgres и локального object storage |
| Пилот до ~50 активных пользователей | 4 vCPU | 8–12 GB | 120–200 GB SSD | Один сервер, Docker Compose, ежедневный backup |

### 10.2 Рекомендуемый стартовый production для небольшого приватного чата

| Сценарий | CPU | RAM | Disk | Сеть / комментарий |
|---|---:|---:|---:|---|
| До 100 MAU, низкая вложенность файлов | 4–6 vCPU | 12–16 GB | 200 GB NVMe SSD | Подходит для одного узла приложений + БД на том же сервере на старте |
| До 300–500 MAU, с фото и файлами | 8 vCPU | 16–24 GB | 300–500 GB NVMe SSD | Желательно вынести object storage или хотя бы хранить вложения на отдельном volume |
| До 1000 MAU / рост realtime-нагрузки | 8–12 vCPU | 24–32 GB | 500 GB+ NVMe SSD | Желательно отделить БД от application/realtime или использовать managed PostgreSQL |

### 10.3 Декомпозиция по компонентам (если разносить по узлам)

| Компонент | Минимум | Рекомендация |
|---|---|---|
| Reverse proxy + web + API | 2–4 vCPU / 4–8 GB RAM | 4 vCPU / 8 GB RAM |
| PostgreSQL | 2–4 vCPU / 4–8 GB RAM | 4 vCPU / 8–16 GB RAM + быстрый SSD |
| Redis (опционально) | 1 vCPU / 1–2 GB RAM | 2 vCPU / 2–4 GB RAM |
| Object storage / MinIO | 1–2 vCPU / 2–4 GB RAM | 2–4 vCPU / 4–8 GB RAM + отдельный диск под файлы |
| Monitoring | 1 vCPU / 1–2 GB RAM | 2 vCPU / 2–4 GB RAM |

### 10.4 Практические замечания по диску и трафику
- Для чата текст сам по себе занимает мало места; основной рост диска дают фото, видео и документы.
- Если пользователи активно отправляют изображения, разумно сразу закладывать запас по хранилищу минимум 200–300 GB либо использовать внешнее object storage.
- Под базу данных и под пользовательские файлы желательно иметь разные каталоги/volume; в идеале — разные диски или managed storage.
- NVMe SSD предпочтительнее обычного HDD/сетевого медленного диска, особенно для Postgres и индексов.
- Для регулярного backup нужно предусмотреть отдельное место хранения вне основного сервера.

## 11. ПО и системные зависимости
- Linux server (Ubuntu LTS либо аналогичный стабильный дистрибутив)
- Docker Engine + Docker Compose
- Nginx / Traefik как внешний reverse proxy
- PostgreSQL 15+
- Node.js LTS для backend и frontend build/runtime
- Резервное копирование: `pg_dump` / wal-based backup / snapshot-based backup по выбранной схеме

## 12. Требования к мобильному этапу
- Мобильное приложение должно использовать тот же backend/API и ту же модель авторизации.
- Нужно поддержать push-уведомления, работу со списком чатов, отправку сообщений и вложений.
- UI/UX мобильного клиента должен быть оптимизирован под маленький экран, но без изменения серверной бизнес-логики.
- Архитектурно предпочтителен React Native / Expo либо иной кроссплатформенный стек для ускорения запуска.

## 13. Что не входит в MVP
- Видеозвонки и аудиозвонки.
- Полноценное end-to-end шифрование с отдельным протоколом ключей.
- Исчезающие сообщения, сложная федерация, каналы/паблики.
- Тяжёлая модерация с AI-анализом контента.
- Сложная мультитенантность и корпоративные сценарии.

## 14. Этапы реализации
1. Подготовка архитектуры, схемы данных, UX-макетов и списка экранов.
2. Разработка backend/API/realtime и БД.
3. Разработка web/PWA-клиента.
4. Интеграция хранения файлов, уведомлений и базовой админки.
5. Тестирование, исправление ошибок, пилот на реальных пользователях.
6. Подготовка мобильного клиента на том же backend.

## 15. Итоговая рекомендация
Рекомендуемый старт: 1 production-сервер 4–8 vCPU, 12–16 GB RAM, 200+ GB NVMe SSD, Docker Compose, PostgreSQL, backend/API/realtime, web/PWA и отдельные backup-копии. При росте нагрузки вынести object storage и/или PostgreSQL в отдельный контур.
